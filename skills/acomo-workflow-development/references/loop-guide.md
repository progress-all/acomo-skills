# 開発ループ正本（acomo ワークフローシステム開発）

[SKILL.md](../SKILL.md) から参照される、フェーズ別の入口条件・作業・出口条件（DoD）の正本。
エージェントは各フェーズの出口条件を満たすまで次に進まない。満たせない場合は同じフェーズ内で反復する。

## フェーズ一覧

| Phase | 名称 | 主担当 | 出口条件（要約） |
|-------|------|--------|-----------------|
| 0 | 環境準備 | エージェント | 接続先の疎通と認証が確認できた |
| 1 | ヒアリング設計 | エージェント + ユーザー | draft（definition / dataSchema / policy）にユーザーが概ね合意 |
| 2 | 静的検証 | エージェント（自律） | validate エラー 0・警告は説明済み |
| 3 | 登録 | エージェント | モデル ID を取得しユーザーへ提示 |
| 4 | ウォークスルー | エージェント + ユーザー | 全結末経路を実走し期待どおり終了 |
| 5 | 改善 | エージェント + ユーザー | フィードバック反映後 Phase 2 から再走して green |
| 6 | 引き渡し | エージェント | 運用手順を説明しユーザーが受入 |

---

## Phase 0: 環境準備

**入口**: ユーザーから「acomo でワークフローを作りたい」等の依頼を受けた。

**作業**:

1. **モード判定** — ユーザーが開発者か非開発者か、接続先がローカル開発スタックか運用中テナントかを確認する。
2. **接続確認** — `acomo getCurrentUser` / `acomo getCurrentTenant` で疎通・認証・テナントを確認する（acomo スキル参照）。
   - 認証エラー（終了コード 2）は試行錯誤せず、ユーザーに認証情報の準備を依頼する（acomo スキルの「認証エラー時の扱い」）。
   - ローカル開発スタックでの非対話認証は [walkthrough-testing.md](walkthrough-testing.md) の手順を使う。
3. **権限確認** — モデルの作成・公開にはワークフロー管理者相当の権限が要る。`listWorkflowModels` が通るか確認する。
4. **参照実装の確認（任意）** — 類似業務のモデルが既に対象テナント内にあれば取得して手本にする
   （`acomo listWorkflowModels` → `acomo getWorkflowModel --modelId <ID>`）。無ければ本スキル同梱の
   `fixtures/sample-*.json`（実走確認済みのお手本 4 本 — 一覧は [harness-guide.md](harness-guide.md)）を手本にする。

**既存モデルを手本として読むとき**（テナントを問わず同じ観点で読む）:

- `definition.nodes` — 1 ノード = 1 アクター × 1 作業になっているか、終了イベントが結末ごとに分かれているか。
- `actionPolicies` — 頻出は「起票者本人だけ」（`$user.id == $executor(<開始ノードid>).id`）と
  「ロール保持者だけ」（`$user.roles has "<roleId>"`）の 2 形。ロール ID・グループ ID・ユーザー ID は
  **テナント固有の値**なのでコピーしない — 式の形だけ流用し、ID は対象テナントの実値に差し替える。
- `policy` — 申請タスクは write 中心、承認タスクは read + 承認者入力欄のみ write、終了イベントに read を
  並べて完了後閲覧を許す、という配分を観察する。
- ハーネスで読む — 定義 JSON を保存して `npm run workflow:walkthrough -- <file>` にかけると、
  経路と操作列がそのまま業務フローの説明資料になる。
- 手本は「形」であって「正解」ではない — 業務に合うかは毎回ユーザーに確認する。

**出口（DoD)**: 認証済みで `getCurrentUser` / `listWorkflowModels` が成功する。モードと接続先が確定している。

## Phase 1: ヒアリング設計

**入口**: Phase 0 完了。

**作業**: **acomo-workflow-modeling スキルの対話設計書に完全に従う**。要点のみ:

- 進捗 1 行を毎ターン示す。ヒアリングだけを続けず、仮定を明示して早く draft（`acomo-workflow-model-draft`）を見せる。
- 逆質問は draft を直すための 1〜2 問に絞る。業務を大きく変える点（承認段数・分岐閾値・機密項目）は仮定だけで確定しない。
- definition → dataSchema → policy の順に固める。philosophy.md のセルフレビューを通す。

**非開発者向けの補足**: JSON を見せて合意を取らない。フローを「誰が・何をして・どうなる」の文章と表で説明し、
データ項目は「入力項目一覧（項目名・型・必須・誰が書けるか）」の表で確認する。

**出口（DoD)**: ユーザーが「この流れ・この項目・この権限でよい」と口頭合意した draft がある。

## Phase 2: 静的検証（エージェント自律ループ）

**入口**: draft がある。

**作業**:

```bash
# draft を一時ファイルに保存して検証
npm run workflow:validate -- /tmp/draft.json
```

1. **エラー（E_*）は全件修正**する。修正はエージェントが自律で行い、ユーザーへの報告は結果のみでよい。
2. **警告（W_*）は分類**する:
   - 技術的な改善（`_order` の重複など）→ 直して報告のみ
   - 業務判断が要るもの（「承認タスクに却下経路がない」「この項目はどのタスクでも編集できない」など）→ ユーザーに 1〜2 問で確認
3. 修正のたびに再実行し、**エラー 0 になるまで登録に進まない**。

**出口（DoD)**: `workflow:validate` がエラー 0（exit code 0）。残る警告は説明・合意済み。

## Phase 3: 登録

**入口**: Phase 2 green。

**作業**（acomo スキルの CLI 規約に従う）:

```bash
# ドラフト登録（policy まで含めて登録する。未確定項目が残る場合はその旨をユーザーに明示）
acomo createWorkflowModel '{"name":"...","description":"...","definition":{...},"dataSchema":{...},"policy":{...}}'
```

- 返却された **modelId を控える**（以降の save / publish / start で使う）。
- 既存モデルの改善なら `saveWorkflowModel --modelId <ID>` を使う（新規作成しない）。
- **Phase 2 を飛ばして登録しない** — サーバー側のスキーマ検証を通っても実行時に壊れるモデルがある
  （実測: `conditions` のない `parallelJoin` は登録も公開も通るが、開始時に 500 になり
  **save / delete も 500 になってモデルが API から削除できなくなる**。復旧は DB 直接削除のみ。
  ハーネスの `E_JOIN_NO_CONDITIONS` が登録前に検知する）。

**出口（DoD)**: modelId が取得でき、`getWorkflowModelWithLatestModelHistory --modelId <ID>` で登録内容が draft と一致する。

## Phase 4: ウォークスルー（実走テスト）

**入口**: モデルが登録済み。

**作業**: 正本は [walkthrough-testing.md](walkthrough-testing.md)。要点:

1. **計画生成** — `npm run workflow:walkthrough -- /tmp/draft.json` で経路別のテスト計画（CLI コマンド列 + UI 手順 + サンプルデータ）を得る。
2. **公開** — `acomo publishWorkflowModel --modelId <ID>`（公開しないとプロセスを開始できない）。
3. **実走** — 計画の**全経路**（承認・却下・分岐・並列の各結末）を実走する。
   - 同伴モード: ユーザーに画面操作を acomo-ui の言葉で案内し、エージェントは CLI（`getProcessWithNodeActions` 等）で状態を並走確認する。もしくはエージェントが CLI で全経路を通し、ユーザーには要所の画面確認だけ依頼する。
   - 各遷移後に**現在ノードが計画どおりか**を確認し、記録する。
4. **証跡** — 経路ごとに「開始プロセス ID・実行アクション列・到達した終了ノード名」を記録し、ユーザーへ提示する。

**アクター（実行ユーザー）の注意**: actionPolicies でタスクごとに実行者が制限される。経路上のタスクを実行できる
ユーザー（ロール・グループ）を事前に確認し、必要なら複数ユーザーの認証を切り替えて実走する。

**出口（DoD)**: 全経路が期待した終了ノードに到達。権限どおりに「操作できない人には操作が出ない」ことも最低 1 ケース確認。

---

## Phase 5: 改善（フィードバックループ）

**入口**: ウォークスルーまたは運用でフィードバックが出た。

**作業**:

1. フィードバックが **definition / dataSchema / policy / actionPolicies のどこに効くか**を短く説明する（全作り直しに見せない）。
2. draft を差分更新し、**Phase 2（静的検証）→ Phase 3（`saveWorkflowModel`）→ Phase 4（影響経路の再ウォークスルー）** を再走する。
   - 影響が特定の経路に閉じるなら、再ウォークスルーはその経路 + 承認のハッピーパスに絞ってよい。
   - dataSchema のキー削除・改名は既存プロセスへの影響をユーザーに確認してから行う。
3. 再公開（`publishWorkflowModel`）まで行い、バージョンが上がったことを確認する。

**出口（DoD)**: 反映後の Phase 2〜4 が green で、ユーザーが変更点を受け入れた。

---

## Phase 6: 引き渡し

**入口**: ユーザーが受け入れた。

**分岐**: 操作面が acomo 標準 UI / CLI で足りない（独自の申請画面・ダッシュボード等が要る）場合は、
ここで引き渡さず **カスタムアプリトラック（A1〜A4）** へ続ける — 正本: [custom-app-guide.md](custom-app-guide.md)。
その場合の最終引き渡しは A4 が担う。

**作業**:

1. **運用手順の説明** — 業務利用者向けに「プロセスの開始 → 入力 → 提出 → 承認」の画面手順を acomo-ui の言葉で説明する（スキル名・パスは出さない）。
2. **管理手順の説明** — モデルの修正が必要になったときの依頼方法（このループの Phase 5 相当）を伝える。
3. **成果の要約** — モデル名・modelId・バージョン・テスト済み経路の一覧を提示する。

**出口（DoD)**: ユーザーが自力で（または依頼ベースで）運用を回せる状態。

---

## アンチパターン

| アンチパターン | 正しい進め方 |
|----------------|--------------|
| ヒアリングを何往復も続けて draft を見せない | 仮定を明示して早く draft を出す（acomo-workflow-modeling の規約） |
| validate を通さず createWorkflowModel する | Phase 2 green が登録の前提 |
| 承認経路だけ実走して完成報告する | walkthrough-plan の全経路を通す |
| エラーを 1 件ずつユーザーに報告して指示を待つ | 技術エラーは自律修正、業務判断だけ質問する |
| フィードバックのたびに新モデルを作り直す | `saveWorkflowModel` で同一モデルを差分更新する |
| 非開発者に JSON・CLI 出力を見せて確認を求める | 業務の言葉・表・画面手順に翻訳して確認する |
